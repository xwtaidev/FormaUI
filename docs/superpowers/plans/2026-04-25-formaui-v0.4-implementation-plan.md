# FormaUI v0.4 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Deliver FormaUI v0.4 as a growth-focused release with an official landing page, reusable marketing blocks, and measurable SEO/performance/conversion baselines while enforcing "FormaUI components only" on landing UI.

**Architecture:** Build v0.4 in six subversions (`v0.4.1` to `v0.4.6`) and keep each checkpoint releasable. Use the existing docs app (`apps/docs`) as the deployment surface for the landing route, compose landing sections from `@formaui/blocks` + `@formaui/components`, and add explicit compliance/quality gates (component-usage check, Lighthouse, analytics event assertions) before GA.

**Tech Stack:** Next.js (App Router), React, TypeScript, Tailwind CSS, FormaUI components/blocks packages, pnpm workspace, Turborepo, Vitest.

## 3. Cross-Version Engineering Rules

- TDD first for blocks and compliance scripts (failing test -> minimal implementation -> pass).
- Landing implementation may use semantic HTML containers, but all interactive UI must come from FormaUI components.
- Avoid parallel creation of ad-hoc "marketing-only primitives" in docs app; shared UI must land in `packages/components` or `packages/blocks`.
- Keep copy/content editable via structured objects and section-level components (avoid hard-coded giant JSX blobs).
- Add/maintain quality gates before visual polish hardening:

```bash
pnpm lint
pnpm typecheck
pnpm test
pnpm build
pnpm registry:build
```

Expected:

```txt
All commands exit with code 0.
Registry JSON is generated for all shipped assets.
```
