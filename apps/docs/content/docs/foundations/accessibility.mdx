# Accessibility

无障碍（Accessibility）不是“上线前补一轮检查”，而是组件设计阶段就必须满足的工程约束。FormaUI 默认以语义化 HTML 与键盘优先交互为基础，但业务接入仍需承担最终可访问性责任。

## 基线原则

- 优先使用原生语义元素（`button`、`a`、`input`、`label`）。
- 所有可交互元素必须可通过键盘访问。
- 焦点可见，不移除 `focus` 样式且对比度足够。
- 文本与关键 UI 对比度满足 WCAG AA。

## 组件级要求

### 表单类组件

- 输入控件必须绑定可感知的 label。
- 错误提示要与输入控件建立可访问关联（如 `aria-describedby`）。
- 必填状态不能仅依赖颜色提示。

### 弹层类组件（Dialog / Drawer / Popover）

- 打开时焦点进入弹层。
- 关闭时焦点回到触发元素。
- 支持 `Esc` 关闭（在业务允许的前提下）。

### 导航与菜单类组件

- 支持方向键与 `Enter`/`Space` 操作。
- 菜单项的可见文本与可访问名称一致。
- 图标按钮必须有可访问名称（`aria-label`）。

## 内容与文案规范

- 链接文本应表达目标，不使用“点击这里”这类弱语义文案。
- 同一页面标题层级有序（`h1` -> `h2` -> `h3`）。
- 状态反馈同时提供颜色与文本提示。

## 发布前检查流程

建议至少执行以下三类检查：

1. **手工键盘走查**
2. **自动化扫描（Axe/Lighthouse）**
3. **关键路径人工阅读测试**（登录、表单提交、关键弹层）

可参考以下命令做基础扫描：

```bash title="bash"
pnpm --filter docs lighthouse
```

## 常见问题

### 只用 Radix 组件是否就“自动无障碍”？

不是。基础交互能力有保障，但业务层仍可能引入问题，例如错误的文案、错误焦点流、缺失描述文本等。

### 为什么看起来“可用”但仍不通过无障碍检查？

视觉可用不等于可访问。屏幕阅读器、键盘用户与低视力用户对结构、焦点和语义的依赖更高。

## 验收清单

- 所有关键流程可通过键盘独立完成。
- 所有 icon-only 按钮具备可访问名称。
- 弹层焦点流正确，关闭后可回到触发点。
- 自动化扫描无阻断级问题（或已记录明确豁免理由）。
