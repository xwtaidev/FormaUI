# Reuse Blocks Guide

## Authoring Flow

Use this flow when editing `/landing`:

1. Update copy/data structures in `app/landing/content.ts`.
2. Keep section containers thin and map data into `@formaui/blocks` props.
3. Run checks:

```bash
pnpm --filter docs lint:landing-components
pnpm --filter docs test
pnpm --filter docs build
```

## Reuse Strategy

When requirements change:

- If only text or links changed, edit `content.ts`.
- If section structure changed but fits existing block APIs, update only container mapping.
- If new reusable UI is needed, add/update assets in `@formaui/blocks` or `@formaui/components` first.

## Copy Update Checklist

- Keep CTA labels and URLs aligned with product docs paths.
- Keep section anchor IDs stable unless tests are updated intentionally.
- Keep inventory proof numbers aligned with current registry output.

## Do / Don't

Do:

- Reuse `HeroCta`, `FeatureGrid`, `StatsStrip`, `LogoCloud`, `FaqAccordion`, `FinalCta`.
- Keep landing section IDs and order deterministic.
- Keep non-visual logic minimal in section components.

Don't:

- Add direct imports from third-party UI kits in landing files.
- Inline hard-coded content blobs in multiple section files.
- Fork one-off block variants in docs app when shared blocks are the right home.
